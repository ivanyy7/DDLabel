## 1. Цель и контекст

- **Проект:** DDLabel — приложение для маркировки продуктов в небольшом кафе: голосовой ввод → разбор фразы → расчёт срока годности по справочнику → печать этикетки.
- **Пользователь:** сейчас один; позже возможно 4–10.
- **Объём:** до ~100 этикеток в смену.
- **Принтер:** Xprinter XP-365B, термо, этикетки 30×20 мм, USB (Этап 1), далее Bluetooth (Этап 3).
- **Язык:** только русский (голос, названия, даты, команды).

**Содержимое этикетки:** название продукта; дата и время изготовления (начало); срок годности до (дата и время окончания). Штрихкоды не нужны. Перспектива: выбор формата (только «срок годности до» или «изготовление + срок годности»).

**Правило расчёта срока годности:** дата/время изготовления + срок из справочника (часы/дни) → рассчитанное время окончания **минус 5 минут** → это «срок годности до» на этикетке.

**Сценарий:** пользователь говорит, например: «Бекон слайс, изготовление вчера в 18:10». Приложение: распознаёт речь; выделяет название продукта и относительные дату/время; знает текущие дату и время; вычисляет дату/время изготовления; по справочнику берёт срок хранения; считает «срок годности до» (минус 5 мин); подставляет в шаблон и печатает.

---

## 2. Справочник сроков хранения (отдельный пункт)

- **Назначение:** продукт (название) → срок хранения (часы или дни). Редактируется **только в приложении** (без правок кода при смене поставщиков).
- **Редактирование в приложении:** раздел «Справочник» — добавление, изменение, удаление записей; сохранение.
- **Загрузка/обновление из файла:** кнопка «Загрузить из файла» / «Обновить справочник». Формат: CSV (обязательно), колонки: название продукта, срок, единица (дни/часы); опционально — импорт из xlsx. После нажатия «Сохранить» или «Обновить справочник» новые данные вступают в силу.
- **Хранение:** локально (JSON или SQLite на ПК; на мобильном — по выбранному стеку).

## 3. Этапы разработки

**Этап 1 — ПК-прототип (браузер + локальный сервис)**  
Голос (Web Speech API, русский) → отправка текста на локальный сервис → разбор фразы (парсер или ИИ): продукт + относительные дата/время → текущие дата/время известны приложению → поиск срока в справочнике → расчёт «срок годности до» (минус 5 мин) → подстановка в шаблон этикетки → печать на Xprinter XP-365B (USB). Отладка на ПК. Интерфейс: голос, отображение результата, раздел «Справочник», настройки (в т.ч. светлая/тёмная тема). Критерий готовности: сказал фразу — этикетка напечаталась корректно.

**Порядок работ внутри Этапа 1 (последовательные мини-этапы):**

1. **Каркас и печать по шаблону** — поднять Node.js (Express) и фронт (браузер); шаблон этикетки по docs/label-format.md; формирование ESC/POS и отправка на Xprinter XP-365B по USB. Тестовые данные (название, дата/время изготовления, срок до) зашиты в запрос или в кнопку. Результат: по нажатию кнопки печатается этикетка с заданным текстом.
2. **Парсер фразы и расчёт срока** — из текста фразы извлечь название продукта и относительные дату/время изготовления («вчера 18:10» и т.п.); по текущей дате/времени вычислить дату/время изготовления; по справочнику (пока один файл или захардкоженный список) взять срок хранения; вычислить «срок годности до» (минус 5 мин). API: на вход — текст фразы, на выход — структура (продукт, дата/время изг., дата/время до). Результат: разбор и расчёт работают, при необходимости подставляются в шаблон и печать из п.1.
3. **Голосовой ввод** — в браузере Web Speech API (русский); кнопка «Голос» → распознавание → отправка текста на сервис → вызов парсера и расчёта (п.2) → печать (п.1). Результат: сказал фразу — этикетка печатается (справочник пока тестовый).
4. **Справочник: хранение и интерфейс** — хранение справочника (JSON или SQLite); API: CRUD по продуктам (название, срок, единица); на фронте раздел «Справочник»: таблица, добавление, редактирование, удаление, сохранение. Результат: справочник редактируется в приложении и используется при разборе фразы вместо тестовых данных.
5. **Импорт и обновление справочника из файла** — загрузка CSV (колонки: название, срок, единица); опционально xlsx. Кнопки «Загрузить из файла» / «Обновить справочник»; после подтверждения новые данные вступают в силу. Результат: справочник можно обновить одним файлом.
6. **Настройки и тема** — раздел «Настройки» в интерфейсе; переключатель светлая/тёмная тема; при необходимости — выбор принтера или параметры этикетки. Результат: интерфейс с настройками и переключением темы.
7. **Тесты и полировка** — покрытие тестами: парсер дат, расчёт срока, API справочника; печать по возможности. Исправление замечаний, проверка полного сценария. Результат: Этап 1 закрыт по критериям из раздела 6.

**Этап 2 — Telegram-бот**  
Голос/текст в Telegram (ПК или телефон) → бот разбирает фразу, считает срок по справочнику (логика как в Этапе 1) → отправляет задание на печать на ПК (локальный мост) → печать на XP-365B (USB). Справочник и настройки: через веб-интерфейс Этапа 1 или отдельный контур (зафиксировать в задаче этапа). Критерий готовности: с ПК и с телефона через Telegram можно напечатать этикетку.

**Этап 3 — Мобильная версия (Android)**  
Приложение на Android: голос (и позже кнопки/шаблоны для частых продуктов) → та же логика разбора и расчёта → печать по Bluetooth на XP-365B. Справочник: редактирование и при необходимости синхронизация/импорт. Критерий готовности: полный цикл с телефона без ПК.

## 4. Стек (Этап 1)

- **Фронт:** браузер — **React**; Web Speech API для голоса.
- **Локальный сервис на ПК:** **Node.js (Express)** — приём текста, разбор фразы, работа со справочником (CRUD, импорт CSV/xlsx), формирование задания на печать.
- **Разбор фразы:** старт — парсер правил + парсер русских дат/времени; опционально позже — ИИ по API.
- **Печать:** библиотека ESC/POS для Node (escpos / node-thermal-printer); Xprinter XP-365B по USB.
- **Справочник:** хранение в JSON или SQLite; импорт CSV (и опционально xlsx).

## 7. Изменения в плане (печать на XP-365B)

- Изначально для печати планировался язык команд **ESC/POS** (через библиотеку `escpos` / `escpos-usb`) и чековый режим принтера.
- В процессе эксплуатации выяснилось, что принтер уже используется с телефона через родное приложение в **режиме печати этикеток**, и переключать его в чековый режим нежелательно.
- Принято решение: для DDLabel перейти на «родной» **язык этикеток TSPL/TSPL2** (или совместимый режим), отказаться от завязки на ESC/POS и реализовать собственный драйвер печати этикеток в Node.js.
- Этап 1 дополняется мини-проектом: выделить слой «драйвер принтера», описать формат этикетки 30×20 мм на TSPL и обеспечить печать по одной и той же схеме как с ПК (браузер), так и в перспективе из Telegram/Android.

## 8. Детализация Этапа 2 (работа с телефона через Vercel и PWA)

**Схема:** Телефон → Vercel (API) → телефон → Bluetooth → принтер. ПК дома, принтер и телефон на работе.

**Принятые решения:**

1. **Web/PWA** — основной путь реализации.
2. **Стратегия:** для проверки идеи и быстрого старта — Web/PWA. Если Bluetooth с принтером не заработает — тогда нативное приложение.
3. **Офлайн:** текстовый ввод — парсер и справочник уже есть, остаётся только кэш. Голосовой ввод для офлайна — позже.

### 8.1. Деплой на Vercel

- Бэкенд (парсер, справочник, API) на Vercel.
- Фронт на Vercel.
- Один URL для ПК и телефона.

### 8.2. PWA (для телефона)

- manifest.json и Service Worker.
- «Добавить на главный экран».
- Открытие без адресной строки, на весь экран.
- **Уточнение:** PWA — для телефона (иконка на главном экране, запуск как приложение). На ПК достаточно открывать сайт в браузере.

### 8.3. Раздел «Печать» в Настройках

- Отдельная кнопка или блок в разделе «Настройки».
- Описание **режимов работы с телефона** (не «ПК vs телефон»):
  - **Офлайн:** телефон + Bluetooth-принтер + локальный справочник на устройстве; ввод только текстом (голос без интернета недоступен).
  - **Онлайн:** телефон + Bluetooth-принтер; справочник синхронизируется с Vercel при наличии интернета; доступны голос и текст.
- Цель блока: пользователь понимает, когда он в офлайне/онлайне и откуда берётся справочник.

### 8.4. Синхронизация справочника

- **Цель:** обеспечить работу с телефона без интернета — парсер и печать по Bluetooth используют локальную копию справочника.
- Локальное хранение на устройстве (localStorage или IndexedDB).
- При наличии интернета — загрузка и обновление справочника с Vercel.
- Автообновление локальной копии при появлении подключения.

### 8.5. Печать по Bluetooth с телефона

- Генерация TSPL на телефоне.
- Отправка на принтер по Bluetooth (Web Bluetooth или Capacitor).

### 8.6. Офлайн-режим

- **Парсер в JS на клиенте** — та же логика разбора фразы и расчёта срока, выполняется в браузере (входит в бандл PWA, при установке кэшируется). Нужен для работы без запросов к серверу.
- **Поведение по умолчанию:**
  - При наличии интернета: запрос к API (парсер и справочник на сервере Vercel); парсер на устройстве не вызывается.
  - При отсутствии интернета или ошибке сети (таймаут, сбой): автоматический **fallback** — парсер на устройстве + локальный справочник (8.4) → расчёт срока → сборка TSPL → печать по Bluetooth.
- **Офлайн только с текстовым вводом:** голос (Web Speech API) без интернета не работает; для офлайна — ввод фразы текстом.
- **Опционально:** в Настройках — кнопка или переключатель «Работать офлайн»; при включении приложение сразу использует локальный сценарий (парсер + локальный справочник), не обращаясь к серверу.
- **Сообщение пользователю при fallback:** при автоматическом переходе на локальный сценарий из-за сбоя сети показывать сообщение вроде: «Нет интернета или нестабильная сеть. Используется локальный режим. Для постоянной работы без сети включите «Работать офлайн» в настройках.»

### Отдельные решения (вне этого этапа)

- Web/PWA выбран; нативное приложение — запасной вариант, если Web Bluetooth не заработает.
- Импорт справочника из файла — уже отложен.

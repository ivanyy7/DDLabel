# План проекта DDLabel

## Роль

Ты — опытный мидл/сеньор разработчик. Действуй в этой роли при выполнении плана: ответственные решения, понятная архитектура, тесты, сроки, без костылей в ключевых местах.

## 1. Цель и контекст

- **Проект:** DDLabel — приложение для маркировки продуктов в небольшом кафе: голосовой ввод → разбор фразы → расчёт срока годности по справочнику → печать этикетки.
- **Пользователь:** сейчас один; позже возможно 4–10.
- **Объём:** до ~100 этикеток в смену.
- **Принтер:** Xprinter XP-365B, термо, этикетки 30×20 мм, USB (Этап 1), далее Bluetooth (Этап 3).
- **Язык:** только русский (голос, названия, даты, команды).

**Содержимое этикетки:** название продукта; дата и время изготовления (начало); срок годности до (дата и время окончания). Штрихкоды не нужны. Перспектива: выбор формата (только «срок годности до» или «изготовление + срок годности»).

**Правило расчёта срока годности:** дата/время изготовления + срок из справочника (часы/дни) → рассчитанное время окончания **минус 5 минут** → это «срок годности до» на этикетке.

**Сценарий:** пользователь говорит, например: «Бекон слайс, изготовление вчера в 18:10». Приложение: распознаёт речь; выделяет название продукта и относительные дату/время; знает текущие дату и время; вычисляет дату/время изготовления; по справочнику берёт срок хранения; считает «срок годности до» (минус 5 мин); подставляет в шаблон и печатает.

---

## 2. Справочник сроков хранения (отдельный пункт)

- **Назначение:** продукт (название) → срок хранения (часы или дни). Редактируется **только в приложении** (без правок кода при смене поставщиков).
- **Редактирование в приложении:** раздел «Справочник» — добавление, изменение, удаление записей; сохранение.
- **Загрузка/обновление из файла:** кнопка «Загрузить из файла» / «Обновить справочник». Формат: CSV (обязательно), колонки: название продукта, срок, единица (дни/часы); опционально — импорт из xlsx. После нажатия «Сохранить» или «Обновить справочник» новые данные вступают в силу.
- **Хранение:** локально (JSON или SQLite на ПК; на мобильном — по выбранному стеку).

---

## 3. Этапы разработки

**Этап 1 — ПК-прототип (браузер + локальный сервис)**  
Голос (Web Speech API, русский) → отправка текста на локальный сервис → разбор фразы (парсер или ИИ): продукт + относительные дата/время → текущие дата/время известны приложению → поиск срока в справочнике → расчёт «срок годности до» (минус 5 мин) → подстановка в шаблон этикетки → печать на Xprinter XP-365B (USB). Отладка на ПК. Интерфейс: голос, отображение результата, раздел «Справочник», настройки (в т.ч. светлая/тёмная тема). Критерий готовности: сказал фразу — этикетка напечаталась корректно.

**Этап 2 — Telegram-бот**  
Голос/текст в Telegram (ПК или телефон) → бот разбирает фразу, считает срок по справочнику (логика как в Этапе 1) → отправляет задание на печать на ПК (локальный мост) → печать на XP-365B (USB). Справочник и настройки: через веб-интерфейс Этапа 1 или отдельный контур (зафиксировать в задаче этапа). Критерий готовности: с ПК и с телефона через Telegram можно напечатать этикетку.

**Этап 3 — Мобильная версия (Android)**  
Приложение на Android: голос (и позже кнопки/шаблоны для частых продуктов) → та же логика разбора и расчёта → печать по Bluetooth на XP-365B. Справочник: редактирование и при необходимости синхронизация/импорт. Критерий готовности: полный цикл с телефона без ПК.

---

## 4. Стек (Этап 1)

- **Фронт:** браузер — React или простой HTML/CSS/JS; Web Speech API для голоса.
- **Локальный сервис на ПК:** **Node.js (Express)** — приём текста, разбор фразы, работа со справочником (CRUD, импорт CSV/xlsx), формирование задания на печать.
- **Разбор фразы:** старт — парсер правил + парсер русских дат/времени; опционально позже — ИИ по API.
- **Печать:** библиотека ESC/POS для Node (escpos / node-thermal-printer); Xprinter XP-365B по USB.
- **Справочник:** хранение в JSON или SQLite; импорт CSV (и опционально xlsx).

(Фронт: React или простой JS — уточнить в начале Этапа 1.)

## 5. Качество, сроки, интерфейс

- **Тесты:** обязательное покрытие (парсер дат, расчёт срока, логика справочника, API сервиса; печать — по возможности).
- **Сроки:** первый рабочий прототип (Этап 1) — 1–2 дня; следующие дни — доработка до полного функционала Этапа 1 по плану.
- **Интерфейс:** простой и понятный; отдельный раздел «Настройки»; переключатель светлая/тёмная тема. Подход как к коммерческому заказу: надёжно, предсказуемо, без временных решений в ключевых местах.

---

## 6. Критерии готовности по этапам

- **Этап 1:** голосовая фраза на русском → корректные название, дата/время изготовления и «срок годности до» на этикетке; справочник редактируется и обновляется из файла; печать на XP-365B по USB; есть настройки и тема.
- **Этап 2:** печать этикетки по запросу из Telegram (с ПК и с телефона) через тот же принтер на ПК.
- **Этап 3:** печать с Android по Bluetooth на XP-365B без использования ПК.

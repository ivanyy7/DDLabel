## Ключевые шаги и история

- *Старт* — созданы PLAN.md (пока пустой), PROJECT_MAP.md, PROJECT_LOG.md. Идёт обсуждение будущего проекта перед написанием плана.

- План оформлен в PLAN.md: роль (мидл/сеньор), цель и контекст, справочник (отдельный пункт), этапы 1–2–3, стек, качество и сроки, критерии готовности.
- Создан PROMPT_PLAN.md: промт для составления плана, обязательные вопросы, роль.
- Выбран стек: Node.js (Express) для локального сервиса; зафиксировано в PLAN.md.
- Добавлены .gitignore (Node), CI (.github/workflows/ci.yml), все файлы (план, карта, лог, промт, правила) закоммичены и запушены на GitHub.
- Добавлен справочник по принтеру: docs/printer-xp365b.md (выжимка из руководства — драйверы, Android Print Label, USB/Bluetooth, характеристики, устранение неполадок). Карта обновлена.
- Фронт зафиксирован: React (удобство таблицы справочника и структуры с первого дня). PLAN.md и PROJECT_MAP.md обновлены.
- В PLAN.md добавлен порядок работ внутри Этапа 1: семь последовательных мини-этапов (каркас и печать → парсер и расчёт → голос → справочник → импорт из файла → настройки и тема → тесты и полировка).
- Зафиксированы формат этикетки (docs/label-format.md), встроенные шрифты принтера (конкретный шрифт/начертание — уточнит пользователь), ссылки на драйверы и ПО принтера (Google Drive) в docs/printer-xp365b.md.
- День 1, финал сессии — этап планирования завершён. Все решения зафиксированы в PLAN.md, PROJECT_MAP.md, docs/. Готовность к старту реализации Этапа 1.
- День 2 — мини-этап 1 (каркас и печать по шаблону): созданы server/ (Express, escpos, escpos-usb) и client/ (React, Vite). Реализован шаблон этикетки в labelBuilder.js по docs/label-format.md (название по центру, даты/время в две колонки, символ ∞). API POST /api/print принимает productName, madeAt, expiresAt; при отсутствии принтера возвращает 503. Фронт: кнопка «Печать тестовой этикетки» с тестовыми данными, прокси на сервер. README обновлён — как запускать сервер и фронт.
- День 2, финал сессии — мини-этап 1 закрыт и зафиксирован. Интерфейс переведён на порт 3000 (vite.config.js: port 3000, host: true для стабильного доступа). Читаемость: в карточке заданы явные цвета (тёмный текст на светлом фоне), чтобы без расширения Dark Reader текст был читаем. Проверено: при отключённом принтере корректно показывается «Принтер не найден». Записи внесены в LOG и MAP.
- День 2 — установка драйвера Windows (Xprinter_2025.1, Seagull Driver Wizard); для доступа DDLabel по USB — Zadig, замена драйвера на WinUSB (принтер в списке как Printer-80). Ошибка «usb.on is not a function» устранена: в server/package.json добавлен override «usb»: «1.8.8». Добавлен отладочный GET /api/printers, логирование ошибок печати. Локальная печать тестовой этикетки отлажена в коде, но фактическая печать на Xprinter XP-365B пользователем ещё не проводилась — требуется отдельная проверка на реальном принтере.
- День 2 — мини-этап 2 (парсер фразы и расчёт срока) завершён. Добавлены server/phraseParser.js (извлечение продукта и относительных дата/время: вчера, сегодня, позавчера, время ЧЧ:ММ), server/shelfLife.js (тестовый справочник сроков в часах, расчёт «срок годности до» минус 5 мин). API: POST /api/parse (фраза → productName, madeAt, expiresAt), POST /api/print расширен — принимает phrase, разбирает и печатает. Фронт: поле ввода фразы, кнопки «Только разобрать» и «Разобрать и напечатать», вывод разобранных данных; читаемость поля ввода (фон #fff, цвет текста #1a1a1a). Проверено: разбор и печать по фразе работают.
- День 2 — мини-этап 3 (голосовой ввод) закрыт. Фронт: Web Speech API (ru-RU), кнопка «Голос»; распознанный текст подставляется в поле и отправляется на разбор и печать (общая функция sendPhraseToPrint). При отсутствии поддержки (не Chrome/Edge) показывается сообщение. Проверено: голосом фраза распознаётся, этикетка печатается.
- День N — справочник сроков: хранение (shelfStorage), API CRUD, раздел «Справочник» на фронте (добавление, список продуктов в модалке, редактирование, удаление, алиасы, «На этикетке» в карточке редактирования); колонка «На этикетке» убрана из таблицы списка.
- День N — парсер: опорная дата по московскому времени (getMoscowNow, makeMoscowDate в phraseParser.js); распознавание «N дня/дней назад» — цифрой и словами (один–десять); удаление «изготовлен/изготовлена» из названия продукта. Парсер при этом работает нестабильно: «два дня назад» и дополнительные названия продуктов могут срабатывать не всегда — требуется доработка в новом чате.
- День N — таблица «Список продуктов»: ресайз колонок ползунками (ширины в rem в state, colgroup); сохранён вариант для проработки в docs/UI-размеры-таблица-продуктов.md (фиксированные ширины в px). Мини-этап на текущем уровне зафиксирован; продолжение (парсер, размеры) — в новом чате.
- День N, 00:00 — доработка окна «Список продуктов»: сначала фиксированные ширины колонок в px, затем перевод на адаптивный вариант (процентные ширины столбцов, увеличение max-width модалки, лёгкий @media для узких экранов). Долго разбирались с разметкой: граница столбца «Срок» и выравнивание кнопок «Изменить / DEL», влияние padding внутри flex-ячейки и table-layout: fixed; в итоге добились аккуратного вида без горизонтального скролла на обычном рабочем разрешении.
- День N, 00:00 — режимы парсера и жёсткий шаблон фразы: вынесен общий движок `parsing/core/phraseEngine` с режимом `template` (ИИ-режим заложен на будущее), добавлен `parsing/template/templateParser` с жёстким шаблоном «<продукт> срок с <дата> с <время>». Парсер научен понимать дни до 31-го и месяцы как цифрами, так и словами (в т.ч. «ноль третьего»), а также часть форм слов для минут («десяти» и т.п.); добавлена гибкость поиска продуктов в `shelfStorage` по наборам слов, чтобы «Сыр Россия» и «Россия сыр» находили одну запись.
- День N, 15:10 — эксплуатационная доработка парсера: жёсткий шаблон упрощён до «<продукт> <ДД> <ММ> <ЧЧ> <ММ>» без обязательного слова «срок» и предлогов «с»; добавлена нормализация форматов дат и времени («6.03» → «6 03», «15:10»/«15.10» → «15 10»), а также очистка лишних «с» как на клиенте, так и на сервере.
- День N, 15:20 — голосовой ввод: увеличено (а затем чуть уменьшено) окно приёма фразы для удобного набора с паузами (~10–11 секунд), фраза из распознавания сразу нормализуется и отправляется на печать; проверены реальные варианты типа «сыр Россия 8 03.18 10», «бекон слайс 6 03.18 10» — этикетки печатаются корректно, формат признан рабочим базовым сценарием.
- День N, 15:30 — переосмысление печати: выяснилось, что принтер XP-365B уже используется с телефона через родное приложение в режиме этикеток, а попытка печати из DDLabel в ESC/POS-режиме не даёт результата (сервер видит устройство, но принтер молчит). Принято архитектурное решение: отказаться от ориентации на чековый ESC/POS и перейти на «родной» язык этикеток принтера (TSPL/TSPL2) с отдельным драйвером в Node.js.
- День N — реализован драйвер TSPL: `server/tsplDriver.js` (buildTsplLabel, buildTsplTestDates), печать по USB сырыми командами TSPL; этикетка 30×20 мм, кодировка CP866. Сервер переведён на TSPL (doPrint вызывает tsplDriver), печать по фразе и тестовая печать работают.
- День N — визуальный редактор макета: превью этикетки в клиенте с ползунками (размер шрифта, сдвиг X/Y, жирность, сжатие/растяжение по ширине) по элементам (название, даты, ∞, время), чекбоксы видимости, сохранение шаблонов в localStorage; калибровочные поля дат, кнопка «Тестовая печать (без фразы)», POST /api/test-print с передачей всех параметров макета на сервер для калибровки.
- День N — калибровка TSPL: даты печатаются шрифтом "2" (легче "0"/"1"), ползунок сжатия -6…6 для тонкой настройки ширины; toSy с Math.floor вместо round, чтобы размеры 37 и 38 px не давали скачок позиции (2→3). Превью: шрифт OpenSans заменён на Teko (Google Fonts), явно задан для макета этикетки.
- День N, финал сессии — печать по TSPL идёт, тестовая печать и печать по фразе работают; результат по читаемости/качеству этикетки не достигнут — «что то печатает, но это так и осталось что то». В следующем чате: довести калибровку до приемлемого вида или зафиксировать текущее состояние и план дальше.

- День N — калибровка тестовой печати дат: полностью переработан интерфейс TSPL-настроек. Вместо CSS-конвертаций — прямые параметры TSPL (font, sx, sy, x, y, density, speed) передаются в принтер без преобразований. Реализован dropdown «Ширина символа» (все комбинации встроенных шрифтов × масштабы sx, отсортированные по ширине символа в точках). Добавлены ползунки DENSITY (0–15) и SPEED (1–5) для управления жирностью/толщиной линий. Подобраны рабочие настройки для дат: шрифт 3 (16×24), sx=1, sy=2, x=20/150, y=75, DENSITY=1, SPEED=4. Результат: чёткие, не жирные цифры дат, хорошо читаемые на этикетке 30×20 мм. Настройки зафиксированы как дефолтные.
- День N — полная этикетка в тестовой печати: добавлены время (ЧЧ.ММ) и название продукта с теми же настраиваемыми TSPL-параметрами (шрифт, sx, sy, x, y). Итоговая раскладка: название вверху (шрифт 3, sy=2, x=20, y=25), даты по центру (шрифт 3, sy=2, x=18/145, y=80), время внизу (шрифт 1 sx=2, sy=2, x=18/145, y=135). DENSITY=1, SPEED=4. Результат признан рабочим — этикетка готова к использованию с голосовым вводом.
- День N, 12:00, финал сессии — сессия отладки этикетки в Debug-режиме завершена. За ~2 часа решено то, что не удавалось в обычном режиме за целый день: UI-настройки шрифтов не доходили до принтера (захардкоженные значения в TSPL-драйвере). Полностью переработан подход: прямые TSPL-параметры вместо CSS-конвертаций, dropdown ширины символа (WIDTH_PRESETS), ползунки density/speed, настраиваемые блоки для названия + дат + времени. Дефолтные настройки зафиксированы. Следующий шаг: интегрировать эти настройки в buildTsplLabel (рабочая печать по голосу).
- День N, 12:30 — интеграция калиброванного макета в рабочую печать: переписан buildTsplLabel под те же параметры, что и в панели TSPL (название 3-й шрифт, даты 3-й шрифт, время 1-й шрифт sx=2; DENSITY=1, SPEED=4). Проверено: фразы вида «сыр Россия 10 03 11 10» и голосовой ввод дают этикетку с тем же макетом, что и тестовая печать (название сверху, даты и время внизу) — результат признан эксплуатационным.
- День N, 12:00 — доработка фронта: кнопка «Голос» переведена в режим тумблера (включён/выключен), введён параллельный голосовой и печатный режимы: при включённом голосе ввод идёт только с микрофона, при выключенном — только вручную в поле.
- День N, 12:05 — пакетный ввод шаблонов: голосом можно последовательно проговаривать несколько шаблонов с соединительным «и», слово «ок» запускает немедленную печать, при паузе 7 секунд после последнего шаблона печать стартует автоматически; в текстовом режиме несколько шаблонов вводятся столбиком (каждая строка — отдельный шаблон), кнопка «На печать» печатает их по очереди.
- День N, 12:10 — интерфейс ввода фразы: поле расширено по ширине, стало многострочным и тянется вниз за нижнюю границу (до ~50 строк); кнопки упорядочены в ряд «Голос / На печать / Разобрать / Сброс», выровнены по ширине, подписи укорочены до одного слова для быстрого чтения.
- День N — верхнее меню с тремя вкладками: Главная (ввод фразы), Справочник (сроки), Настройки (тема + превью этикетки). Вкладки вынесены над главным окном, выровнены по правому краю. Переключатель темы Светлая/Тёмная в Настройках; выбор сохраняется в localStorage.

- День N — планирование Этапа 2: обсуждение альтернатив (Telegram vs Vercel + два приложения), Web/PWA vs нативное приложение, офлайн-режим, схема «телефон → Vercel → Bluetooth → принтер». Контекст: ПК дома, принтер и телефон на работе. PLAN.md дополнен разделом 8 «Детализация Этапа 2» (деплой Vercel, PWA для телефона, раздел «Печать» в Настройках, синхронизация справочника, Bluetooth, офлайн). Принятые решения: Web/PWA основной путь; стратегия — PWA сначала, нативное приложение если Web Bluetooth не заработает; офлайн — текстовый ввод (парсер + кэш), голос позже.

Печать подтверждена: 4× Грязные овощи + 1× Грязные фрукты.

## Технические решения и грабли

- Шрифты: только встроенные принтера (ESC/POS); конкретный размер/начертание пользователь укажет позже.
- Фронт: React (таблица справочника, структура).
- Порядок работ: семь мини-этапов внутри Этапа 1 — выполнять последовательно, каждый с явным результатом.
- Интерфейс: порт 3000; в Vite задано host: true для надёжного доступа к localhost.
- Читаемость: в карточке и в поле ввода фразы явно заданы цвет текста (#1a1a1a) и светлый фон, чтобы без расширений темы текст оставался читаемым.
- Печать по USB (Windows): после установки драйвера Windows принтер «занят» системой; для DDLabel (escpos-usb) нужен Zadig — замена драйвера на WinUSB. Инструкция в docs/printer-xp365b.md.
- Ошибка «usb.on is not a function»: пакет usb (зависимость escpos-usb) в новых версиях несовместим с escpos-usb. Решение: в server/package.json задать "overrides": { "usb": "1.8.8" }, затем npm install.
- Мини-этап 2: парсер опирается на ключевые слова (вчера/сегодня/позавчера) и шаблон времени ЧЧ:ММ; продукт — остаток фразы после удаления этих частей. Справочник сроков пока захардкожен в shelfLife.js (часы); «срок годности до» = изготовление + срок − 5 минут.
- Мини-этап 3: голос — Web Speech API (SpeechRecognition/webkitSpeechRecognition), язык ru-RU. Поддержка в Chrome, Edge; при отсутствии API показывается сообщение «Голосовой ввод не поддерживается в этом браузере».
- Парсер дат: опорная дата «сегодня» берётся по московскому времени (Intl.DateTimeFormat, timeZone: Europe/Moscow); «N дня/дней назад» — по цифре или по словам один–десять. Парсер пока нестабилен — темы для доработки: надёжное распознавание «два дня назад» и поиск по дополнительным названиям (алиасам).
- Таблица «Список продуктов»: первый вариант — ширины колонок в state (rem) и ресайз ползунками; фиксированный вариант в px описан в docs/UI-размеры-таблица-продуктов.md. В текущей версии, после долгих попыток добиться аккуратного выравнивания столбца «Срок» и кнопок, перешли на адаптивную таблицу с процентными ширинами столбцов и небольшими @media-правками; важно учитывать, что изменение padding внутри flex-ячейки не двигает границу столбца при table-layout: fixed — для выравнивания нужно работать с ширинами колонок и общей шириной модалки.
- Поиск в справочнике: простое сравнение строк оказалось недостаточно — реальный ввод/голос даёт варианты «Сыр Россия», «Россия сыр» и т.п. Рабочий компромисс: нормализуем строку (регистр, пробелы) и сравниваем наборы слов (productName/aliases ↔ фраза), оставляя старый подстрочный поиск только как резерв.
- TSPL: координаты и масштабы в точках (203 dpi); toSy — из пикселей UI в sy (масштаб по высоте), использован Math.floor(v/15), чтобы 37 и 38 px не давали скачок 2→3 и «прыжок» этикетки вверх/вниз. Сжатие по ширине: слайдер -6…6 → sx = sy + stretch (clamp 1–8). Шрифт дат "2" для более лёгкого начертания. Превью: шрифт Teko (Google Fonts) для макета этикетки.
- Калибровка TSPL (Debug-сессия): корень проблемы — CSS-настройки UI вообще не влияли на печать, шрифт и масштаб были захардкожены в tsplDriver.js. Решение: полный отказ от CSS→TSPL конвертаций; все параметры (font, sx, sy, x, y, density, speed) передаются напрямую из UI в TSPL-команды. WIDTH_PRESETS — программно сгенерированный массив всех комбинаций шрифт×sx для dropdown с точной шириной символа в точках. DENSITY (0–15) управляет нагревом печатающей головки (жирность), SPEED (1–5) — скоростью (чем быстрее, тем тоньше линии). Оптимум для XP-365B: DENSITY=1, SPEED=4.
- Debug-режим в Cursor: принципиально отличается от обычного — ИИ обязан сначала логировать и доказать причину, потом чинить. В обычном режиме ИИ угадывает по коду и сразу правит, что часто приводит к зацикливанию. Рекомендация: планирование и написание кода — обычный режим; при зависании на баге — сразу переключать в Debug.
- Тема и кнопки: глобальное правило `html[data-theme="light"] button` и `.card button` переопределяют стили вложенных элементов. Для читаемости кнопок темы и вкладок нужны селекторы с `:not(.theme-toggle-active)` и `html[data-theme="dark"] .app-tab` с `!important`. Для зелёных кнопок (Голос, Одиночн) — селектор `.card .mode-toggle-btn.mode-toggle-active` для переопределения в светлой теме.
- Этап 2: вместо Telegram выбран Vercel + Web/PWA. Web Speech API — облачный, не работает офлайн; для офлайн-голоса нужны Vosk/Whisper или нативное приложение. Офлайн-режим: сначала текстовый ввод (парсер + кэш справочника).
- Голос «ок»: при «ок» отдельным результатом — задержка 450 мс перед печатью (pendingVoiceTemplatesRef), чтобы успеть принять «грязные фрукты» и т.п.; «ок» не пишется в поле; ошибка только при not-allowed (микрофон).
- Пакетная печать «N штук»: parseTemplateQuantity распознаёт цифры и слова (два–пятьдесят); resolveTemplatesWithQuantity обрабатывает «штуки» на отдельной строке — берёт число из конца предыдущего шаблона.
- Двойной режим, позиционирование текста: время (y=135) упиралось в нижний край этикетки (160 точек) и обрезалось. Все координаты двойного режима сдвинуты вверх: название y=22, даты y=77, время y=132. Подобрано итерациями (−10 → −5 → +2).
- Голос и labelMode: stale closure — при переключении «Одиночн» после старта голоса sendPhraseToPrint видел старый режим. Решение: labelModeRef, обновляемый в useEffect; sendPhraseToPrint читает labelModeRef.current.
- Голос и «четыре штуки»: распознавание может выдать «24» вместо «четыре» и разбить «24 штуки» на два сегмента (число прилипает к шаблону, «штуки» — отдельно). Решение: при новой сессии, если новый сегмент — количественный суффикс (содержит «штук»), склеиваем с последним накопленным шаблоном; плюс защитный слой в resolveTemplatesWithQuantity.
- Выравнивание дат: время (ЧЧ:ММ) смещено по y вниз (101 вместо 80), чтобы нижний край цифр совпадал с нижним краем даты (ДД.ММ) на этикетке. Подобрано эмпирически: промежуточные значения 96→97→99→101, итоговое y=101.
- Пробелы в текстовом вводе: onChange у textarea вызывал splitTextTemplates(), который делал .trim() на каждую строку — пробелы обрезались сразу при наборе. Исправлено: onChange просто делает setPhrase(e.target.value); splitTextTemplates вызывается только при печати.
- Кнопки DEL в светлой теме: глобальное правило `html[data-theme="light"] .card button { background-color: #1a1a1a !important }` перекрывает любой цвет кнопок внутри .card. Для исключений нужны более специфичные селекторы с `!important` (например `.shelf-actions-cell .shelf-del-btn`).
- Перезагрузка страницы на кнопках: `<button>` без `type="button"` по умолчанию имеет `type="submit"` — браузер воспринимает его как submit-кнопку и перезагружает страницу. Всегда явно указывать `type="button"` на не-submit кнопках в React.
